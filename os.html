What is the difference between Semaphore and Mutex and in what situations are each used?

Semaphore and mutex are both synchronization mechanisms used in concurrent programming to control access to shared resources.
A mutex (short for mutual exclusion) is a locking mechanism that allows only one thread to access a resource at a time. It provides exclusive access and is typically used to protect critical sections of code or shared data structures. When a thread acquires the mutex, it can proceed with the critical section, and when it releases the mutex, another thread can acquire it and proceed.
On the other hand, a semaphore is a generalized version of a mutex that can allow multiple threads to access a resource, up to a specified limit. Semaphores can be used to control access to a pool of resources, for example, limiting the number of simultaneous connections to a database or the number of threads that can access a file.
In summary, a mutex provides exclusive access to a resource by allowing only one thread at a time, while a semaphore allows multiple threads to access a resource, up to a specified limit. The choice between the two depends on the specific requirements of the concurrent program and the type of resource being protected.

How can a deadlock occur in a multi-threaded program and how can it be avoided?

Deadlock in a multi-threaded program occurs when two or more threads are blocked, each waiting for the other to release a resource, resulting in a situation where no progress can be made. Deadlocks typically occur due to a combination of four conditions:
1. Mutual Exclusion: Resources cannot be shared; only one thread can exclusively hold a resource at a time.
2. Hold and Wait: Threads hold resources while waiting for others.
3. No Preemption: Resources cannot be forcibly taken from the threads holding them.
4. Circular Wait: There is a circular chain of two or more threads each waiting for a resource held by the next thread in the chain.
To avoid deadlocks, several strategies can be employed:
1. Avoidance: Use careful resource allocation and ordering to ensure that circular wait conditions cannot occur.
2. Detection and Recovery: Implement timeout mechanisms or resource-reclamation policies to break deadlocks when they occur.
3. Prevention: Allow preemption of resources from threads to prevent deadlocks from occurring.
4. Resource ordering: Establish a strict ordering of resource requests, ensuring that threads always request resources in the same order to prevent circular waits.
By applying these strategies and designing concurrent programs with careful consideration of resource interactions, deadlocks can be alleviated or prevented.
